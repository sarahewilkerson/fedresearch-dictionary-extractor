## 1. Context

User-driven validation across 27 stratified Army PDFs surfaced **80% auto-precision** on extracted glossary entries (698 good / 168 bad). Manual spot-check of 62 entries from 11 small docs confirmed the auto-classifier's accuracy and surfaced **five recurring extractor failure modes**:

1. **Continuation-line splitting** — Wrapped def lines at left margin treated as new terms (PAM 350-58 #4, #6, #9, #11, #15, #20, #23, #26, #29, #34 — ~30% of that doc fragmented).
2. **Asterisk-prefix terms split mid-term** — `*engineer work line` becomes term=`*engineer`, def=`work line A coordinated…`.
3. **Section structure markers absorbed as terms** — `SECTION`, `SECTION I`, `SECTION II`, `Glossary-4`.
4. **Page-footer text bleeding into entry definitions** — `AR 190-55 • 30 October 2023` appearing as the def for an adjacent entry.
5. **End-of-glossary back-cover artifacts** — `UNCLASSIFIED`, `PIN 060296-000`, `U.S Army` appearing as terms.

Admin review queue catches noise in production, but production user-facing search precision must be ≥90% for v1 launch. **PR1.2 wheel publication is gated on this PR.**

The OLD extractor at `/Users/mw/code/Projects-from-Library/JAGC Projects/FedResearch_Dictionary_Creator` had bold-flag detection, footer Y-filtering, and 5-layer term validation. The port lost bold detection + footer filter; this PR restores them with safety improvements informed by adversarial review.

## 3. Fixes

### Fix A — Per-line bold-flag gate for new-term detection (bugs #1, #2)

**Codex finding 1 fix:** Decision is per-candidate-line, not per-page. Each candidate left-margin line is classified one of three ways:

- **STRICT:** First span is bold (`is_span_bold` true) → new term.
- **ALL-CAPS heuristic** (acronym-section pattern): First span non-bold AND first word matches `^[A-Z][A-Z\-]{1,14}$` (2-15 chars, all upper or hyphenated) AND full line length ≤ 60 chars AND no `.` in line's first 30 chars (rules out continuation lines whose first word happens to be capitalized — e.g., "Furthermore, the…"). Handles acronym sections in AR 600-20, FM 6-02. Codex iter-2 #1.
- **Otherwise:** Treat as continuation of current term's def.

Implementation in `extractors/glossary.py`. Codex iter-3 #5: the prose, snippet, and tests must use **one shared helper** so the gate predicate is unambiguous:

```python
# Helper at module top
_ACRONYM_FIRST_WORD_RE = re.compile(r"^[A-Z][A-Z\-]{1,14}$")

def _looks_like_acronym_term_line(line_text: str) -> bool:
    """Codex iter-3 #5: ALL-CAPS heuristic — first word is acronym-shaped (2-15 chars,
    upper or hyphenated), AND full line is glossary-term-shaped (≤60 chars,
    no period in first 30 chars — rules out 'Furthermore, the…' continuations)."""
    if len(line_text) > 60:
        return False
    if "." in line_text[:30]:
        return False
    first_word = line_text.split(maxsplit=1)[0] if line_text else ""
    return bool(_ACRONYM_FIRST_WORD_RE.match(first_word))

# Per-line gate (replaces the simple is_bold check)
first = line_spans[0]
first_x = first["bbox"][0]
in_term_col = first_x < term_col_threshold
full_line_text = " ".join(s["text"] for s in line_spans).strip()
is_bold = text_utils.is_span_bold(first["span"])
is_acronym_line = _looks_like_acronym_term_line(full_line_text)

if profile.enable_bold_gate:
    is_new_term_line = in_term_col and (is_bold or is_acronym_line)
else:
    is_new_term_line = in_term_col   # legacy X-only fallback

if is_new_term_line:
    # new-term path
else:
    # continuation path (or skip if no current_term)
```

Unit-test the helper directly with: ✓ `"AIT"`, ✓ `"USINDOPACOM"`, ✓ `"DODIN"`, ✗ `"Furthermore, the term is used"`, ✗ `"A coordinated boundary line for engineers."`, ✗ `"U.S. Army"` (period in first 30), ✗ `"FOREIGN INTELLIGENCE SURVEILLANCE COURT"` (>60 chars).

**Per-doc fallback rate metric:** During validation re-run, log `bold_gate_rate = bold_admit / left_margin_lines` per doc, exposed via `analyze_pdf()["metadata"]["glossary_parser"]["bold_gate_rate"]` (Codex iter-3 #7). If a doc has `bold_gate_rate < 30%` AND has detected zero entries, it likely doesn't preserve bold flags reliably. The plan does NOT auto-disable the gate on those docs (too dangerous); they get flagged for manual inspection in `scorecard.json` `flagged_for_review` list (Codex iter-3 #8 — the "admin queue" claim was unsupported; flagged_for_review is the actual mechanism inside this repo, not a downstream backend feature).

**Fix #2 (asterisk split) — bold-span walk:** When line passes new-term gate, term-extraction walks consecutive bold-or-allcaps spans (Codex finding 5). Stop conditions: (a) hit a non-bold-non-allcaps span, OR (b) hit a span containing terminal punctuation `.!?` followed by space + lowercase, OR (c) hit a span starting with `(` (citation marker).

```python
term_spans = [first]
def_start_idx = len(line_spans)
for j in range(1, len(line_spans)):
    sp = line_spans[j]
    sp_word = sp["text"].split(maxsplit=1)[0] if sp["text"] else ""
    sp_is_term_style = (
        text_utils.is_span_bold(sp["span"])
        or bool(re.fullmatch(r"[A-Z][A-Z\-]{1,}", sp_word))
    )
    if not sp_is_term_style:
        def_start_idx = j; break
    if re.search(r"[.!?]\s+[a-z]", sp["text"]):
        def_start_idx = j; break
    if sp["text"].startswith("("):
        def_start_idx = j; break
    term_spans.append(sp)

term_text = " ".join(s["text"] for s in term_spans).strip()
def_remainder = " ".join(s["text"] for s in line_spans[def_start_idx:])
```

**Leading-`*` policy:** Preserve `*` in extracted term (it's a meaningful glossary marker in Army doctrine — denotes new/changed term). The `normalize_term` function in `normalize.py` already strips leading punctuation for the normalized form, so search isn't affected.

**Inline-split compatibility:** Existing `split_re` (line 107) operates on the joined `term_text`. After bold-span walk produces a multi-word `term_text`, run the existing inline-split logic against it unchanged — preserves `Term. Definition…` handling.

### Fix D — Narrow end-of-glossary patterns (bug #5)

Codex finding 2: UNCLASSIFIED is unsafe as a hard range terminator. Treat UNCLASSIFIED at term-position as **rejection** (Fix C handles it). For range termination, require **proximity to PIN** OR **back-cover position** (last 3 pages of the doc).

In `extractors/glossary.py`, modify `find_glossary_page_range`:

```python
# Detect a "back-cover-like" page: contains both UNCLASSIFIED + PIN,
# or is one of the last 2 pages of the document AND contains UNCLASSIFIED
def _is_back_cover_marker(page_text: str, page_idx: int, total: int) -> bool:
    """Codex iter-2 #3: require BOTH PIN proximity AND last-3-pages position.
    UNCLASSIFIED alone (anywhere) is NEVER a terminator — many normal pages
    have UNCLASSIFIED in their footers."""
    has_pin = bool(re.search(r"\bPIN\s+\d{4,}", page_text))
    in_last_3 = (total - page_idx) <= 3
    return has_pin and in_last_3
```

Then the merge function itself:

```python
# Same-page continuation merge (residual fragments that slipped past Fix A).
# Constraints: only adjacent entries from the SAME page; only when the prior
# def doesn't end with terminal punctuation AND the next term starts lowercase
# (a real new term in Army glossaries always starts uppercase, allcaps, or '*').
def _merge_same_page_continuations(entries: list[dict]) -> list[dict]:
    if len(entries) < 2:
        return entries
    out = [entries[0]]
    for e in entries[1:]:
        prev = out[-1]
        if (
            prev["pdf_page_index"] == e["pdf_page_index"]
            and not re.search(r"[.!?]\s*$", prev["definition"])
            and re.match(r"^[a-z]", e["term"])
        ):
            prev["definition"] = (prev["definition"] + " " + e["term"] + " " + e["definition"]).strip()
            continue
        out.append(e)
    return out
```

## 6. Hard 30%

- **Bold-flag preservation in OCR'd PDFs.** All 27 sample PDFs are `_OCR` suffix. If most pages have unreliable bold flags AND the ALL-CAPS fallback doesn't catch their term style, the fix gains nothing. **Mitigation:** validation re-run reports `bold_gate_rate` per doc; if any high-volume doc shows <30% bold-admit rate, escalate before PR1.2.
- **Bold-or-allcaps gate may still drop legitimate mixed-case lowercase-start terms** in OCR'd docs (rare but exists — e.g., "the term healthcare" from AR 12-15, which the user already confirmed as a continuation fragment, but other docs may have legit lowercase headwords). Regression-recall check (§5d) catches this aggregate; per-doc audit during scorecard re-run catches individual cases.
- **Continuation-merge in Fix E could mis-merge** when a glossary entry intentionally has a lowercase-starting follow-on term in the same page (rare in Army regs but possible in alphabetically-grouped sections). **Mitigation:** the constraint requires the prior def to NOT end with terminal punctuation — most legitimate adjacent terms are separated by complete-sentence defs.
- **Footer Y-zone of 88%** is a heuristic. Validated by: (a) per-page count of footer-filtered lines should be ≤2 per page; if any page filters >2 lines, threshold is too aggressive — log + investigate.
- **AR 135-100 exception precedent.** This is a documented acceptance for one specific doc, NOT a pattern for general low-quality docs. Adding a second exception in the future requires explicit re-review and threshold redefinition.
