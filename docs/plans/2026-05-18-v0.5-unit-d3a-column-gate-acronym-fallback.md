# v0.5 Unit D-3-A: Column-Gate Fallback for Section-I Acronym Lists

**Date:** 2026-05-18
**Branch:** `feat/2026-05-18-v0.5-d3a` (off main post-D-2)
**Roadmap parent:** `docs/plans/2026-05-18-v0.5-roadmap-definition-extraction.md`
**Findings memo:** `docs/plans/2026-05-18-v0.5-unit-d2-findings.md` (Class P-1)
**Parallel sibling:** D-3-B (top-of-page running-header filter) — independent parser surface; can land in either order.

## Context

D-2's findings memo identified **Class P-1** as the dominant parser failure across the 11 verified parser-dead-end docs (4 primary + 6 hybrid). The pattern:

Docs with a Section I (Acronyms and Abbreviations) section laid out as:
```
ADP   Army doctrine publication
ADRP  Army doctrine reference publication
AO    area of operations
```

The trace shows `_looks_like_acronym_term_line` correctly identifies these as acronym-shaped (30+ true-results per doc). But the gate is `is_new_term_line = in_term_col AND (is_bold OR is_acronym_line)` — the `in_term_col` check short-circuits to False because:

1. The section header `SECTION I — ACRONYMS AND ABBREVIATIONS` sits at the leftmost x position on the page.
2. `min_x = min(line[0]["bbox"][0] for line in valid_lines)` — the section header's x becomes `min_x`.
3. `term_col_threshold = min_x + TERM_COL_MARGIN` (30 points).
4. Acronym lines start at `min_x + ~40-60` (indented past the threshold).
5. `in_term_col = first_x < threshold` → **False**.
6. Line classified as continuation → but no `current_term` is set → silently dropped.

## Root cause

The `min_x` computation assumes the lowest-x line on the page IS a term-column line. When that line is actually a Section header (visually different layout from glossary content), the min_x is misleading and the column-gate becomes too strict.

## Approach

**Relax the column-gate when `is_acronym_line=True`,** with a safety bound to prevent over-acceptance.

```python
# Current (extractors/glossary.py:521):
in_term_col = first_x < term_col_threshold
...
is_new_term_line = in_term_col and (is_bold or is_acronym_line)

# Proposed (corrected per Codex iter-1 #1):
in_term_col = first_x < term_col_threshold
# D-3-A relaxation: acronym-shaped lines pass column gate within a
# wider band — admits BOTH bold and non-bold acronyms past the strict
# 30pt term-col but within reasonable distance of min_x.
in_acronym_band = first_x < (min_x + ACRONYM_COL_MARGIN)  # ACRONYM_COL_MARGIN: TBD via margin sweep (§Pre-implementation)
is_new_term_line = (in_term_col and (is_bold or is_acronym_line)) \
                   or (in_acronym_band and is_acronym_line)
```

The new branch fires ONLY for acronym-shaped lines (existing `_looks_like_acronym_term_line` check) outside the strict term-column but within the wider acronym band. Bold-but-not-acronym lines are unaffected (require the existing strict band per the first branch). **Bold AND acronym-shaped lines outside the strict band are now also admitted** — this fixes the iter-1 #1 finding.

**Why this works:** acronym lines have a very specific shape (≤14 chars, all caps, valid prefix). False-positive risk is bounded by `_looks_like_acronym_term_line`'s existing constraints AND the new wider-but-still-bounded x-band.

**Alternative considered (rejected):** exclude section-header lines from min_x computation. More complex (need to identify section headers reliably) and could have cascading effects on the column-threshold for legitimate term-column lines.

## Hard 30%

### H1. False-positive risk on prose lines starting with acronyms (revised per Codex iter-1 #2)
Lines like `"DOD policy requires the following..."` or `"U.S. Army forces shall..."` start with acronym-shaped words but are body prose. The existing `_looks_like_acronym_term_line` may accept them if total length ≤60 chars AND no period in first 30 chars (for non-dotted acronyms).

**Tested mitigation:** new unit tests include realistic prose-line negatives:
- `"DOD policy requires..."` (37 chars; `DOD` is acronym-shaped; no period in first 30) — currently passes `_looks_like_acronym_term_line` (bug surface).
- `"AR 600-20 applies"` (currently passes the acronym detector because prefix is acronym).

The plan acknowledges these as **pre-existing false-positive surface in `_looks_like_acronym_term_line`** — NOT introduced by D-3-A. D-3-A widens the column band but doesn't change which lines are classified as acronym-shaped. **If false-positive recovery exceeds 5% of recovered entries, defer to a separate "acronym-detector tightening" unit (D-3-D).**

### H1b. (was H1) Body-text false positives outside the band
Lines further right than `min_x + ACRONYM_COL_MARGIN` are still excluded by the band check. So body prose deeply nested in column-text won't be admitted. This bounds the false-positive surface.

### H2. Regression risk on 31-doc validation set
The relaxation changes the gate for ANY acronym-line outside the strict term-col band but within 80pt. Docs in the validation set may have acronym-shaped headers (e.g., publication-name running-headers) that previously failed `in_term_col` and now pass. **Mitigation:** run 31-doc golden-output regression. Any newly-extracted terms get reviewer-approval via `accepted-deviations.yaml` per D-1's pattern.

### H3. ACRONYM_COL_MARGIN value choice (resolved via S1 sweep)
Value chosen empirically via S1 margin sweep (40/60/80/100/120). Smallest value meeting recovery target AND keeping validation-set deltas ≤1 wins. Document the chosen value + sweep evidence in the code comment.

### H4. Interaction with X-only fallback (revised per Codex iter-1 #4)
The X-only fallback runs when `bold_preservation_rate < 0.10` AND `len(fallback_entries) > len(glossary_entries)`. With D-3-A landed:
- If the visual-path produces entries (the new acronym-band gate fires), `glossary_entries` grows. Fallback path now needs more entries than before to win.
- If `glossary_entries == 0` post-D-3-A (D-3-A didn't help), fallback still fires if it has any entries.
- **New risk:** fallback's "all lines in term-col are new terms" cascade could produce noisier output that count-wins over D-3-A's cleaner output.

**Mitigation:** S1's margin sweep records both `glossary_entries` and `entries_after_dedup` per doc, AND `metadata.glossary_used_legacy_fallback`. Acceptance asserts: if `glossary_used_legacy_fallback=True`, the chosen entries must include the expected acronyms from S2 ground truth. If fallback wins by count but loses by correctness, this is a tracked failure mode and falls back to "report it; defer cleaner-vs-noisier-output to D-3-E if observed".
