# v0.4.0 — Glossary Header Tolerance + Lookback Extension

**Date:** 2026-05-17
**Branch:** `feat/2026-05-17-v0.4-header-and-lookback`
**Target release:** `fedresearch-dictionary-extractor` v0.4.0
**Downstream:** `EXTRACTOR_VERSION` bump `army-v2.1.0` → `army-v2.2.0` in `apps/backend/src/definition-extraction/extraction-worker.service.ts:60`

## Problem

The v0.3.0 re-extraction wave produced zero entries for 177 of 1546 Army Pubs
docs (11.5%) despite each having a glossary section. Empirical sampling of 4
representative failures (`AR 735-5`, `AR 420-1`, `PAM 73-1`, `ATP 3-21.10`)
isolates the failure into the `find_glossary_page_range` stage in 3 of 4
cases. Two distinct sub-causes:

1. **Header-phrase variant** — the v0.3.0 regex set in
   `profiles/army.py:62` requires whole-line `^\s*Glossary\s*$` or
   `^\s*Section\s+II\s*[-–—]?\s*Terms\s*$` etc. The phrase
   `Glossary of Terms` (used by 124 of 177 failing docs, 70%) does not
   match any pattern.
2. **Lookback distance too short** — `MAX_GLOSSARY_LOOKBACK_PAGES = 30` in
   `extractors/glossary.py:54` scans only the last 30 pages of the document.
   `PAM 73-1` has its glossary at page 462 of 499 — 37 pages from the end —
   so the backward sweep stops before reaching it.

The fourth failure (`ATP 3-21.10`) is a different class — a broken CMap in
the embedded PDF text layer produces Caesar-shifted bytes that the gibberish
filter correctly rejects. That requires upstream-text fallback (architectural
change) and is **out of scope** for v0.4; tracked separately.

## Root cause

The header-detection logic was tuned against a labeled validation set of
~30 docs in `validation_set/`. Headers in that set are predominantly bare
`Glossary` or Section II variants. Production corpus exposure (1,546 docs in
the May 2026 wave) revealed two header conventions absent from the
validation set, plus a class of long-tail docs (>500 pages) where appendices
push the glossary outside the original 30-page lookback assumption.

## Approach

Two surgical changes, both in the same v0.4.0 release:

### Change 1: Header-pattern extension (`profiles/army.py`)

Add two entries to `glossary_header_patterns`:

```python
r"^\s*Glossary\s+of\s+(?:Terms?|Acronyms?|Abbreviations?)(?:\s+and\s+\w+)?\s*$",
r"^\s*GLOSSARY\s+OF\s+(?:TERMS?|ACRONYMS?|ABBREVIATIONS?)(?:\s+AND\s+\w+)?\s*$",
```

The patterns are still whole-line anchored. The optional trailing
`and \w+` clause catches "Glossary of Terms and Abbreviations" (one observed
variant). `re.IGNORECASE` is already applied at compile time in
`find_glossary_page_range` so the second uppercase pattern is technically
redundant — but it's kept for grep-discoverability and matches the
existing pattern style.

### Change 2: Lookback extension (`extractors/glossary.py`)

Raise `MAX_GLOSSARY_LOOKBACK_PAGES` from `30` to `75`.

Rationale: in the failing set, the deepest glossary observed is at
~page 462 of 499 (PAM 73-1, 37 pages from end). Setting the limit to 75
provides ~2x headroom for unseen variants. The backward sweep already
short-circuits at the first matching page, so cost is bounded by
`75 * O(page_text_length)` worst-case — measured at <1s per doc on
representative samples.

We deliberately do NOT make this unbounded. A full doc scan would
amplify false positives when "Glossary" appears in body text or TOC
references (the regex anchors to line-start but case-insensitive + multiline
mode can match within tables of contents). 75 is the smallest value that
covers the observed failure cohort with margin.

## Hard 30%

- **False-positive header matches.** "Glossary of Terms" might appear inside
  a TOC entry on an earlier page, e.g., a TOC line:
  `Glossary of Terms ...................... G-1`. The regex `\s*$`
  anchor combined with re.MULTILINE *should* reject this because the line
  has trailing content — but `\s*$` accepts trailing whitespace, and PyMuPDF's
  text layout for justified-TOC lines may render the dot-leaders as their
  own visual line separately from the title. **Mitigation:** the new patterns
  require `Glossary of {Terms,Acronyms,Abbreviations}` — full phrase, less
  TOC-like. Verify on a TOC-heavy doc fixture during testing.

- **Existing-doc regression risk.** Some currently-succeeding docs may have
  their glossary at the bare `Glossary` page-N header earlier than another
  page that contains `Glossary of Terms` in body prose. Raising lookback to
  75 increases the surface for this collision. **Mitigation:** the backward
  sweep returns the FIRST match from the end. For an existing-succeeding
  doc, the bare `Glossary` header is already returning a hit at some page
  ≤30 from end — extending the lookback to 75 doesn't change that. The only
  risk is if a body-text occurrence within the new 30-75 page band matches
  the new "Glossary of …" patterns. **Acceptance:** before merge, run v0.4
  against the existing PR-A v0.3.0 candidate-output PDFs (the 27 in
  `validation_set/candidate_output/`) and verify entry counts are
  unchanged.

- **Pattern overreach.** The optional `and \w+` clause is broad — could
  match "Glossary of Terms and References" (where References is a separate
  section). **Mitigation:** acceptable. Worst case is the range extends
  one page too far; the existing end-detection logic (Index/References/
  Appendix patterns) clamps it.

## Blast radius

- **Extractor library**: glossary-detection behavior changes for any doc
  whose glossary header was previously unmatched.
- **Backend**: when EXTRACTOR_VERSION bumps to `army-v2.2.0` and the new
  wheel ships, the operator-UPDATE re-extraction pattern can retarget
  the ~124+ failing-cohort docs.
- **Validation set**: existing labeled docs in `validation_set/` should
  show no behavior change. If they do, the regression is in scope.
- **Other profiles**: none. Only `ArmyProfile.glossary_header_patterns`
  changes; the base class and other profiles are untouched.

## Test strategy

1. **Unit tests** (`tests/test_section_headers.py` already exists; extend):
   - New test: `_glossary_header_matches` parametrized over
     [`Glossary of Terms`, `GLOSSARY OF TERMS`, `Glossary of Acronyms`,
     `Glossary of Acronyms and Abbreviations`, `glossary of terms`].
   - Negative cases: `Glossary of Operations Research` (not in our set —
     should NOT match), inline body prose `... the Glossary of Terms
     contains ...`.
   - Lookback test: synthetic small doc-shape verification of
     `MAX_GLOSSARY_LOOKBACK_PAGES = 75`.

2. **Regression check on PR-A v0.3.0 fixtures**: run the analyzer against
   every PDF in `validation_set/candidate_output/` (27 docs). Diff entry
   counts vs the committed v0.3.0 baseline. Acceptance: zero deltas on the
   labeled set.

3. **Empirical validation on the failure cohort**: run analyzer against
   the 4 sampled failing PDFs (`AR 735-5`, `AR 420-1`, `PAM 73-1`,
   `ATP 3-21.10`). Acceptance:
   - AR 735-5: glossary range found + at least 1 entry produced.
   - AR 420-1: glossary range found + at least 1 entry produced.
   - PAM 73-1: glossary range found (lookback fix) + at least 1 entry.
   - ATP 3-21.10: still 0 entries (out-of-scope failure mode; verifies we
     didn't accidentally fix it via a different path that might produce
     gibberish output).

4. **Full test suite**: `pytest -x` from repo root must pass.

   ```sql
   UPDATE definition_extraction_queue
   SET status='PENDING', extractor_version=NULL,
       failure_reason=NULL, error_snippet=NULL, attempts=0,
       last_attempt_at=NULL, next_attempt_at=NULL
   WHERE status='SUCCEEDED' AND extractor_version='army-v2.1.0'
     AND entry_count = 0
     AND document_id IN (
       SELECT d.id FROM documents d
       WHERE d.extracted_text_status='HAS_TEXT'
         AND d.extracted_text ILIKE '%glossary%'
         AND length(d.extracted_text) > 100000
     );
   ```

## Out of scope

  upstream-text fallback. Separate future plan if/when the cohort size
  justifies the architectural change.
- Multi-column glossary layouts. Defer until v0.4 lands and we re-sample
  the remaining failure cohort.
- Inline-definition extractor (`extractors/inline.py`). Untouched.
- AR 735-5 inspected text format (`Discrepancies\nDisagreement...`).
  Once header detection succeeds, the existing visual-span parser should
  handle this format (bold gate or X-only fallback). Validated empirically.
