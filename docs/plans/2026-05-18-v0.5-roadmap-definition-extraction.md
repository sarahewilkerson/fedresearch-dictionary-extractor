# Roadmap: Improving Definition + Glossary Detection (post-v0.4)

## Context

v0.4 shipped 2026-05-17 and recovered 132 of 177 failing docs (74.6%). The remaining 45 docs are zero-entry-with-glossary, putting the corpus at the 50-doc reopen threshold for the broken-CMap class. The user has confirmed (today): recall is the bigger pain than precision; the residual cohort is worth a v0.5 architectural change; operational coordination pain is worth investing in.

This document is a **meta-plan** — its job is to decompose the work, name invariants, and sequence units. **Per CLAUDE.md `feedback_meta_plan_defers_specifics`, technical specifics (Hard 30%, edge cases, test files, file paths beyond the few needed for orientation) live in per-unit plans, each of which goes through its own `/review-plan`.**

## Operator-confirmed decisions

1. **Unit 2b LLM-judge precision audit:** in-scope (+~$10/month Claude API spend).
2. **Unit 3a Dockerfile-derived `EXTRACTOR_VERSION`:** acceptable; couple wheel version → cache invalidation; opt-out env var for edge cases.
3. **Sequencing:** Unit 0 first; downstream units re-plan after Unit 0 returns.

## Scope of THIS plan

**Only Unit 0 is approved for execution by this plan.** Units 1, 2, 3, 4, 5 are described as candidate units, but each is conditional on Unit 0's output and requires its own per-unit plan. This guards against the iter-2 failure mode where Unit 1 was pre-specified along one branch (text-flow parse stored text) before the underlying assumption (`documents.extracted_text` is clean for broken-CMap PDFs) had been measured.

## Unit 0: Investigation (only approved unit) — S, 1-2 hours

**Goal:** characterize the 45 residual docs across enough dimensions that Unit 1's branch — and whether Units 2 and 3 can run in parallel with it — is informed by data instead of assumption.

### Unit 0 cohort freeze

The 45 docs are defined by this exact prod query, run once at Unit 0 start, with the result set committed as a CSV fixture:

```sql
SELECT q.document_id, d.canonical_id, d.gcs_key
FROM definition_extraction_queue q
JOIN documents d ON d.id = q.document_id
WHERE q.status='SUCCEEDED' AND q.extractor_version='army-v2.2.0' AND q.entry_count = 0
  AND d.extracted_text_status='HAS_TEXT' AND d.extracted_text ILIKE '%glossary%'
ORDER BY q.document_id;
```

Output committed at `validation_set/v0.5-unit0-cohort.csv` (sibling repo) BEFORE measurements run. The CSV freezes the cohort by ID; subsequent corpus changes don't drift the denominator.

### Known vs detected glossary range

Each residual gets TWO glossary-range labels:
- **Detected:** what `find_glossary_page_range` returns (may be `None` for some).
- **Known:** the actual glossary-section page range, established by manual inspection of each PDF's TOC + a quick page-flip. (~30 sec/doc; ~25 min total for 45 docs.)

Measurements 4 and 5 below operate on the **known** range so the load-bearing assumption doesn't quietly leak through unrepresentative page slices.

### Measurements (committed script — `scripts/v0.5-unit0/diagnose.py` in sibling repo, reproducible)

For each of the 45 residuals, the script records:

1. **PDF-source vowel ratio** on `page.get_text("text")` from the known glossary range.
2. **Upstream-text vowel ratio** on `documents.extracted_text` over the known glossary range (page-mapped via measurement 4).
3. **Control-byte rate** in both sources — `\x03`, `\x0c`, `\x0e-\x1f` excluding tab/newline.
4. **Page-boundary preservation in `extracted_text`** — explicit page-delimiter markers (FormFeed, `--- N of M ---`, blank-line cluster, etc.) detectable AND mappable to PDF page indices.
5. **Glossary-header detectability from `extracted_text` alone** — re-run `find_glossary_page_range`-equivalent regex against the stored text; record start/end page (or None).
6. **Whitespace + newline preservation** — ratio of `\n` to character count, distribution of spaces (catches collapsed-whitespace OCR corruption).
7. **Unicode replacement-character rate** — `�` presence (mojibake signal).
8. **Reading-order sanity** — paragraph-level sentence boundary count vs character count (catches two-column interleaving where sentences mid-collapse).
9. **Term/definition shape on a hand-inspected sample** — up to 5 docs per non-empty decision-matrix cell, minimum 10 total across the cohort (selected by deterministic stride over the cohort CSV). Each: tag with `{same-line|colon|dash|bare-newline|acronym-heavy|multi-line-term|two-column|other}`.

### Decision matrix thresholds (numeric branch gates)

Aggregate over the 45-doc result set:

- **Branch A** (text-flow parse stored OCR text) — fires if: ≥30 docs have `text_vowel ≥ 0.30 AND control_byte_rate < 0.001 AND page_boundaries_mappable=True AND glossary_header_detectable_in_text=True`.
- **Branch B** (re-OCR via `/ocr-gcs`) — fires if: ≥30 docs fail Branch A's criteria AND pdf-service `/ocr-gcs` is verified usable (separate spike: <30 min, confirm endpoint, capacity, latency, cost).
- **Branch C** (hybrid) — fires if: split is between 15-29 each way; sub-plan specifies the per-doc routing function.
- **Branch D** (pivot away from broken-CMap framing) — fires if: <15 docs match Branch A AND <15 docs need re-OCR AND ≥10 docs have a recognizable layout issue (multi-column, inline-only definitions, header phrase beyond v0.4's regex set). In this case the next unit is layout-handling, not text-flow.
- **Branch E** (defer entirely) — fires if: <10 docs match any actionable class AND <20 are reasonable candidates for ANY of A/B/D. Memo recommends moving the reopen threshold; no Unit 1 work scheduled.

If results don't cleanly fit a single branch (e.g., 28 match A and 14 match B — between thresholds), the memo flags ambiguity and requests operator decision before per-unit planning.

### Verify (failure gates, not optimistic assumptions)

Unit 0 is done **only when ALL of these hold**:

- `validation_set/v0.5-unit0-cohort.csv` is committed with all 45 doc IDs.
- `scripts/v0.5-unit0/diagnose.py` is committed and reproducible (re-run against the CSV produces byte-identical output).
- The measurement table at `docs/plans/2026-05-18-v0.5-unit0-investigation.md` (sibling repo) has a row for every doc in the cohort CSV. **No "skipped — failed to read" rows are acceptable without a separate "unmeasurable" cell in the matrix** counted explicitly toward Branch E.
- The recommended branch (A/B/C/D/E) satisfies its threshold. If no branch fits, the memo recommends operator review, not "best guess."
- Sibling-repo PR for the memo + cohort + script is merged.
- The user explicitly approves the Unit 1 framing on the basis of the memo.

## Invariants across the roadmap

These bind multiple units. Each per-unit plan must respect them, even though no single unit owns them.

1. **Per-unit plan required.** Each candidate unit gets its own plan-doc in `docs/plans/` (sibling or backend repo as appropriate), through `/review-plan`. No unit executes without an approved per-unit plan.
2. **Cross-unit file overlap.** If two units propose touching the same file (e.g., Unit 1 + Unit 3a both wanting `extraction-worker.service.ts`), the second unit's per-unit plan must name the merge strategy (stacked rebase, integration branch, or merge-before-next). Codex iter-3 #6 caught this contradiction in the prior draft.
3. **Circular evaluation dependency.** Unit 1's acceptance must NOT reference an LLM-judge precision threshold unless Unit 2 has shipped + calibrated the judge first (Codex iter-3 #9). If Unit 1 ships before Unit 2, its acceptance uses a hand-labeled mini-set instead.
4. **Schema-migration-plan-required.** Unit 3b (admin re-extract endpoint with audit table) must include the Prisma migration in its per-unit plan.
5. **Plan-one-approval-path.** Unit 0 → Unit 1 is one approval path: user reviews the memo and approves a single Unit 1 framing. No parallel branches "just in case."
6. **Worktree-first.** All fedresearch backend work in this roadmap uses `git worktree add` (per memory `feedback_use_worktree_for_concurrent_agents`; Unit 3c promotes to CLAUDE.md).
7. **Conditional non-goals.** Multi-column glossary detection and inline-extractor expansion are non-goals **only if** Unit 0 confirms broken-CMap is the dominant residual class. If Unit 0 returns framing D, they become candidate Unit 1 scope (Codex iter-3 #12).

## Out of scope (deliberate non-goals)

- **LayoutLM / ML-based layout extraction.** Engineering investment not justified at corpus size 1,545.
- **LLM-as-primary extraction in the production hot path.** Breaks the `--deterministic` contract from v0.3.0. LLM is **judge-only** (Unit 2b). LLM-as-fallback in the residual class can be considered **only** under Unit 5's go/no-go decision, gated on (a) Unit 2 producing trustworthy precision data, AND (b) explicit operator approval to relax the determinism contract for the residual class only. Without both, LLM stays out of the extraction path entirely.
- **Regenerating `validation_set/candidate-output/` baselines.** Pre-existing 11-doc drift; separate sibling-repo cleanup PR.
- **(Conditional)** Multi-column glossary detection and inline-extractor expansion — non-goals unless Unit 0 framing D fires.

## Why the iter-2 APPROVED was wrong

The iter-2 plan over-specified Unit 1 along the text-flow-parse-stored-text branch and bundled Hard 30%, test strategy, edge cases, and file paths into the meta-plan. Per memory `feedback_meta_plan_defers_specifics`, those specifics belong in per-unit plans where they can be grounded against actual Unit 0 data and current code, not extrapolated from assumptions. Codex iter-3 surfaced 13 issues including: load-bearing assumption never measured (1, 2), missing page-mapping contract (3), circular fallback (4), parser too naive (5), sequencing contradiction (6), eval-harness under-specification (8), circular Unit 1 ↔ Unit 2 dependency (9), and others. The right correction was a fresh meta-plan rewrite (this file) plus explicit "Unit 0 only" approval scope.
