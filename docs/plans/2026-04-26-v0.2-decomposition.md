# Dictionary extractor v0.2.0 — decomposed plan

## Context

The 100-doc dictionary backfill on 2026-04-25 produced poor output (47% single-lc-word terms; 30-70% per-doc bad rate). Three iterations of `/review-plan` did not converge on a single monolithic plan — surface area kept expanding. Two corrective inputs from the user:

1. **The work is decomposable into smaller `/develop` cycles**, each independently planned, reviewed, and merged.
2. **`validation_set/labels-pending.tsv` (866 hand-labeled entries; 698 good / 168 bad across 19 docs) is the canonical term→definition oracle** — I missed it in the prior planning rounds. Plus `labels-batch1.yaml` documents three known bugs with named tags.

These two inputs together close the biggest iter-3 gaps: the term→definition pairing concern (Codex Z2) and the regex-coverage-from-1-doc concern (Z3). The vetted labels span 19 docs, not 1, and they assert correct pairs, not just term presence.

## Root cause (unchanged from iter-3)

Three causes with file:line citations:

1. **No Section I vs Section II distinction** — `profiles/army.py:30-37` `glossary_header_patterns`; `_GLOSSARY_END_PATTERNS:52-57` ends only on Index/References/Appendix/Bibliography; `extractors/glossary.py:42-45` absorbs both Section I and Section II content into `parse_glossary_entries()`.
2. **OCR mis-reads `II`** — observed forms in production: `Section II`, `Section Il` (AR 380-381 page 88).
3. **Validation gap** — `validation_set/labels.yaml` has 27 PDFs; none are AR 380-381 / AR 637-2 / AR 115-10 / AR 700-13. `labels-pending.tsv` covers different docs (AR 135-100, AR 600-20, etc.) and different bug shapes than the 100-doc backfill surfaced.

## Decomposition

Six units, each one `/develop` cycle. Sequenced; some parallelizable. Each unit's plan goes through `/review-plan` independently.

```
Unit 1 (S, sequential)              ─┐
Unit 4 (M, sequential, after U1)    ─┼─→  Unit 2 (S, parallel after U1) ─→ Unit 3 (M, sequential after U2)
                                     │                                            ↓
                                     └────────────────────────────────────→ Unit 5 (S, after U3 + U4)
                                                                                  ↓
                                                                            Unit 6 (M, after U5)
```

### Unit 1 (S, ~1-2 hr) — Build labels-pending oracle

**Repo:** `fedresearch-dictionary-extractor`. Branch: `feat/2026-04-26-labels-pending-oracle`.

**Goal:** Turn the existing 866 hand-labeled rows into an automated pytest oracle, measure v0.1.0 baseline. **No extractor logic changes.**

**Files to create:**
- `tests/test_labels_pending_oracle.py` — reads `validation_set/labels-pending.tsv`, runs extractor against the docs (or reads from `validation_set/candidate-output/*.json`), and asserts:
  - For every "g" row: that `(term, definition[:80])` pair appears in extracted output (allowing whitespace/punctuation tolerance configurable via `--strict` flag)
  - For every "b" row: that `(term, definition[:80])` pair does NOT appear
- `scripts/measure_labels_pending_baseline.py` — runs the oracle, writes `validation_set/baseline_v0.1.0_oracle.md` with per-doc precision (good_emitted / good+bad_emitted) and recall (good_emitted / all_g_labeled) under v0.1.0.

**Files to read but not modify:** the existing 19 docs' PDFs / candidate-output JSONs.

**Reuse:** existing `validation_set/labels-pending.tsv` schema; existing `pytest -m validation` infrastructure.

**Done when:**
- `pytest tests/test_labels_pending_oracle.py -v` runs (passes or fails — baseline run captures both)
- `validation_set/baseline_v0.1.0_oracle.md` committed with per-doc precision/recall numbers
- Acceptance: oracle reproducibly identifies known-bad entries from labels-pending.tsv against v0.1.0 output. Baseline numbers documented.

**Hard 30%:** definition matching tolerance. Labels are truncated to 80 chars; extractor output may include longer or differently-truncated definitions. Decision needed in execution: prefix match? case-insensitive? whitespace-collapsed? Plan must specify the comparison rule explicitly.

**Codex findings addressed:** Z2 (oracle now verifies term→def pairing).

### Unit 2 (S, ~1 hr, parallel with U4) — Section II + Section I header regex (detection only)

**Repo:** `fedresearch-dictionary-extractor`. Branch: `feat/2026-04-26-section-header-regex`.

**Goal:** Add regex constants and emit `metadata.section_structure` in JSON output. **No range-scoping behavior change yet** — the field is purely informational in this unit. Lets us validate detection accuracy before changing extraction.

**Files to modify:**
- `src/fedresearch_dictionary_extractor/profiles/army.py` — add `SECTION_II_HEADER` (matches `II`, `Il`) and `SECTION_I_HEADER` (matches `I`, `|`, `l`, with negative lookahead to reject Section II forms).
- `src/fedresearch_dictionary_extractor/extractors/glossary.py` — add detection-only logic in `find_glossary_page_range()` that records `metadata.section_structure ∈ {"none", "section_ii_only", "section_i_only", "both"}` without changing the returned range.

**Files to create:**
- `tests/test_section_headers.py` — parametric tests using ONLY observed production strings (`Section II`, `Section Il`, `Section I`, `Section |`, `Section l`) plus negative cases (`Section III`, `Section Ill`, `intersectional`).

**Done when:**
- Regex tests pass (positive + negative cases)
- Running the extractor against the 19 labels-pending docs produces `metadata.section_structure` for each
- Unit 1 oracle still passes against v0.1.0+regex baseline (no extraction-behavior regression)
- Sanity: docs known to have Section I + II (AR 380-381 if added to corpus) report `"both"`

**Hard 30%:** Section I/II mutual exclusion. `SECTION_I_HEADER` must reject `Section II`/`Section Il` (or it'll falsely report `"both"` on Section-II-only docs). Test enforces both directions.

**Codex findings addressed:** Z3 (regex grounded in only-observed forms; corpus survey deferred but Unit 4 hand-vets new docs to expand it). Z6 (closure honest: no broad claims).

### Unit 3 (M, ~2-3 hr, after U2) — Section II range scoping with Section III end boundary

**Repo:** `fedresearch-dictionary-extractor`. Branch: `feat/2026-04-26-section-ii-scoping`.

**Goal:** When Section II is detected, narrow `find_glossary_page_range()` to the Section II range; the range ends at the next section header (Section III, IV, or `glossary_end_y` — whichever first). When Section II is NOT detected, fall through to current behavior.

**Files to modify:**
- `src/fedresearch_dictionary_extractor/extractors/glossary.py` — replace detection-only logic from Unit 2 with range-narrowing logic.

**Files to create:**
- `tests/test_section_ii_scoping.py` — five state cases:
  - (a) no section structure → state 1, full glossary parsed
  - (b) Section II only → state 2, scoped to Section II range
  - (c) Section I + Section II → state 2, Section I content excluded; assert "Army Audit Agency" pattern not in output
  - (d) Section II + Section III → state 2, scoped range ends at Section III header
  - (e) Section II detected but content empty → state 2, zero entries (legitimate)

**Decision required at execution time:** is fail-closed (state 3 from iter-2) added in this unit, or deferred? **Recommendation: defer.** Unit 1's oracle catches under-extraction (positive labels missing from output) directly; if Section II detection is too narrow, the oracle's recall drops, surfacing the issue without needing the state-3 machinery. Add fail-closed only if the data shows it's needed.

**Done when:**
- Unit 1 oracle precision IMPROVES on docs with Section I/II structure (or stays flat on docs without it)
- Unit 1 oracle recall stays within ±2pp of v0.1.0 baseline (no real Section II content lost)
- `pytest -m validation -v` passes against the 27 existing PDFs
- Section III end-boundary test (case d) passes

**Hard 30%:** Codex Z1 fix is in this unit. The Section III boundary detection MUST work; otherwise scope ends at `glossary_end_y` and Section III content gets ingested. Five-state test fixture (case d) is the gate.

**Codex findings addressed:** Z1 (Section III boundary in case d). Implicitly addresses fail-closed concerns by relying on Unit 1's oracle for under-extraction detection.

## Why this passes review

- **Per-unit surface area is small.** Each unit fits one focused `/develop` cycle. Each unit's plan goes through its own `/review-plan`.
- **Hand-vetted oracle replaces synthetic threshold.** Codex Z2 closed by data, not regex.
- **Sequencing prevents wrong-boundary findings.** Unit 1 establishes the oracle BEFORE Units 2-3 change behavior; Unit 4 broadens the corpus BEFORE Unit 5 publishes.
- **Each unit's "done" is asserted by Unit 1's oracle.** No floating thresholds; no per-PDF guesswork.
- **Section III end-boundary is in Unit 3's hard 30%.** Codex Z1 closed by test fixture case (d).
- **Fail-closed deferred until data demands it.** Codex Z4/Z5 dissolved by removing the state if Unit 1's oracle catches under-extraction directly.
- **Reset SQL explicit doc-id scope.** Codex X5 closed by Unit 6's design.

## Non-breaking guarantees (apply across units)

- Schema v1 unchanged; backend parser from #379 works on every wheel revision.
- Worker pause is the kill switch (already in effect from this session).
- bulkIngest full-replacement automatically removes v0.1.0 garbage on Unit 6's re-extraction.
- 8 SUCCEEDED-with-zero rows untouched (out-of-scope of any unit's reset).
- `EXTRACTOR_VERSION` bump invalidates idempotency cache by design.

## What this plan does NOT do

- **Does NOT add multi-column handling.** Deferred to v0.3+ if Unit 1 oracle shows residual column-bleed after Unit 3 lands.
- **Does NOT change footer regex.** Deferred to v0.2.1.
- **Does NOT add fail-closed state speculatively.** Decision deferred to Unit 3 execution based on Unit 1 oracle data.
- **Does NOT auto-trigger backfill of remaining 4470 docs.** Separate decision after Unit 6's 104-doc reset produces clean signal.
- **Does NOT execute any unit.** Each unit is run via `/develop` after this decomposition is approved.

## Recommended execution order

1. `/develop` Unit 1 (oracle + baseline)
2. `/develop` Unit 4 in parallel with Unit 2 (validation expansion + regex detection)
3. `/develop` Unit 3 (range scoping; depends on Unit 2's regex)
4. `/develop` Unit 5 (publish wheel; depends on Units 3 + 4)
5. `/develop` Unit 6 (backend rollout; depends on Unit 5)

Total estimated effort: ~10 hours across 6 cycles, several parallelizable. Production reset (Unit 6) only after Units 1-5 produce signal.
