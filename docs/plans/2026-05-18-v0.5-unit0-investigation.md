## Range-match breakdown (the actual failure modes)

### Class 1: Exact range, parser failed (11 docs)

v0.4's `find_glossary_page_range` returned the same start/end pages as a permissive heuristic. The parser then emitted zero entries despite a valid glossary range on clean text. **This is a parser bug, not a header-detection bug.** All 11 are TC/ATP/FM docs with PDF vowel ratio 0.36–0.39 (well into clean text territory). Examples:

- `army:tc:tc-3-22240-medium-machine-gun-2017-04-28` — glossary at pages 194-195 of 202
- `army:tc:tc-3-25150-combatives-2017-03-31` — glossary at pages 398-399 of 408
- `army:tc:tc-21-306-tracked-combat-vehicle-driver-training-2025-04-24` — pages 86-87 of 100

Likely causes (to investigate at Unit 1 plan time): no bold flag preservation + title-case glossary terms + the X-only fallback's all-lines-are-new-term failure mode I described in the v0.4 analysis.

### Class 2: Overlap (21 docs)

v0.4 found A range that overlaps the actual glossary but is mis-bounded — usually v0.4 finds the LAST 2 pages of a 4-30 page glossary because the backward sweep matches the END-of-glossary footer / "Glossary-N" reference first. Examples:

- `atp-3-2110-infantry-rifle-company` — v0.4 picked pages 604-605; actual range pages 580-605 (lost 25 pages of glossary content)
- `tc-3-045-instrument-flight-for-army-aviator` — v0.4 picked 298-299; actual 294-299

Fix direction: validate that the backward-sweep match is a section-header, not a footer/page-number occurrence (font size, position, surrounding context).

### Class 3: Disjoint (6 docs)

v0.4 found a completely wrong location — typically a body-text "see glossary" reference or a per-chapter glossary-of-symbols sidebar. Examples:

- `ar-12-15-joint-security-cooperation` — v0.4 picked page 343 (body); actual glossary at page 21 (TOC-or-prefix)
- `pam-700-127-integrated-product-support` — v0.4 picked page 168; actual range 9-43

Fix direction: same as Class 2 (header validation), plus a tie-breaker preference for earlier match when multiple candidates exist and one looks like a body reference.

### Class 4: Neither (7 docs)

Loose heuristic also failed. Mix of legitimate non-glossary docs (small briefs, <10 pages) and weird-structure docs. Notable: 4 of 7 are very short docs (1, 1, 7, 9 pages) — these almost certainly do not have a glossary section despite the doc's `extracted_text` containing the word "glossary" (probably in a footnote or cross-reference). Should be filtered out at the cohort level next time.

## Branch D scope sketch (NOT YET A PER-UNIT PLAN)

The next v0.5 Unit 1 plan should address two sub-classes:

**D-1: Glossary range detection robustness** (27 docs = Class 2 + Class 3)
- v0.4's backward sweep stops at the first regex match. False-positives include back-cover "Glossary-N" footers, TOC references, and per-chapter sidebars.
- Mitigation candidates:
  - Require the match to be the *only* substantive content on its visual line (filter footer-zone matches via Y position)
  - Validate against a small content-window after the match (real glossary headers are followed by term entries; false ones by body prose)
  - Prefer the EARLIEST glossary-header match in a candidate window, not the latest (the v0.4 backward-sweep choice was wrong for these docs)

**D-2: Exact-range parser failure investigation** (11 docs = Class 1)
- v0.4 found the right pages; parser still emitted 0 entries.
- Per-doc trace needed (similar to today's pre-v0.4 diagnostic) on the 11 to determine whether the failure is: bold preservation absent + acronym gate also failing, X-only fallback's all-new-term cascade, Section II narrowing producing empty range, or something else.

**Not in scope (Branch D):**
- Class 4 docs (7) — these are mostly short docs without real glossaries; should be filtered at cohort-eligibility time, not given a parser path. Recommend a separate cleanup that excludes docs with `total_pages < 30` from the failing-cohort eligibility.
