# v0.5 Unit D-3-B: Top-of-Page Running-Header Filter

**Date:** 2026-05-18
**Branch:** `feat/2026-05-18-v0.5-d3b` (off main post-D-2)
**Roadmap parent:** `docs/plans/2026-05-18-v0.5-roadmap-definition-extraction.md`
**Findings memo:** `docs/plans/2026-05-18-v0.5-unit-d2-findings.md` (Class P-2)
**Parallel sibling:** D-3-A (column-gate acronym fallback) — independent parser surface.

## Context

D-2's findings memo identified **Class P-2** as a contributor to 7 of 11 parser-dead-end docs (1 primary + 6 hybrid). The pattern:

The only line passing the term-column + bold-or-acronym gate is a page running-header at the TOP of glossary pages:
- `"TC 3-22.240 Glossary-1"`
- `"13 January 2017 TC 3-22.6 Glossary-1"`

Format: `[<date>] <TYPE> <pub-number> Glossary-<page-num>`. These pass:
- Length OK (≤60 chars).
- `_looks_like_acronym_term_line` accepts (first word is acronym-shaped: TC, FM, etc.).
- `_validate_term` accepts (length, no terminal period that's too short, not in `invalid_term_patterns`).

Once accepted, the parser awaits continuation lines. Footer is the only line passing the gate on its page, so `current_def_lines=[]` at end-of-page flush → drop.

## Root cause

`parse_glossary_entries` has a footer-zone filter (`extractors/glossary.py:487-505`) but it only matches lines in the BOTTOM 12% of the page (`footer_y_threshold = page.rect.height * FOOTER_ZONE_PCT` where `FOOTER_ZONE_PCT=0.88`). These running-headers appear at the TOP of pages, outside the bottom zone.

Additionally, `is_document_header` (called at line 482) checks lines in the TOP 150pt zone (`HEADER_ZONE_Y`) against `profile.header_patterns`. The current Army `header_patterns` don't include the `<TYPE> <pub#> Glossary-<N>` shape.

## Approach

**Extend `ArmyProfile.header_patterns`** with two patterns matching the observed running-header shapes:

```python
# Current (profiles/army.py:71-78):
@property
def header_patterns(self) -> list[str]:
    return [
        r"HEADQUARTERS\s*\n?\s*DEPARTMENT\s+OF\s+THE\s+ARMY",
        r"By Order of the Secretary of the Army",
        r"SUMMARY of CHANGE",
        r"^\s*(AR|FM|ADP|ATP|TC|PAM|TM|DA\s*PAM)\s+\d+[-–]\d+\s*[•·]\s*\d+\s+\w+\s+\d{4}\s*$",
    ]

# Proposed: add 2 new patterns
@property
def header_patterns(self) -> list[str]:
    return [
        r"HEADQUARTERS\s*\n?\s*DEPARTMENT\s+OF\s+THE\s+ARMY",
        r"By Order of the Secretary of the Army",
        r"SUMMARY of CHANGE",
        r"^\s*(AR|FM|ADP|ATP|TC|PAM|TM|DA\s*PAM)\s+\d+[-–]\d+\s*[•·]\s*\d+\s+\w+\s+\d{4}\s*$",
        # D-3-B: glossary page running-header — "TC 3-22.6 Glossary-1"
        r"^\s*(AR|FM|ADP|ATP|TC|PAM|TM|DA\s*PAM)\s+[\d\.\-–]+\s+Glossary[\s\-–—]+\d+\s*$",
        # D-3-B: dated glossary page running-header — "13 January 2017 TC 3-22.6 Glossary-1"
        r"^\s*\d{1,2}\s+[A-Z][a-z]+\s+\d{4}\s+(AR|FM|ADP|ATP|TC|PAM|TM|DA\s*PAM)\s+[\d\.\-–]+\s+Glossary[\s\-–—]+\d+\s*$",
    ]
```

The new patterns are anchored `^...$` (whole-line, with MULTILINE flag applied at compile time per the existing pipeline). They require:
- Doc-type prefix (AR/FM/ATP/TC/etc.) — preserves the same set as other Army patterns.
- Publication number (digits + dots/hyphens).
- Literal `Glossary-N` suffix.

Optional date prefix in the second pattern (`<day> <month-name> <year>`).

**Existing filter integration:** `is_document_header(line, profile.header_patterns)` (called at `glossary.py:482`) already checks lines in the top-150pt zone. The new patterns hook in automatically — no changes to the call site.

## Integration contract verification (added per Codex iter-1 #1+#3)

**Single consumer.** `header_patterns` is consumed exclusively by `is_document_header` in `extractors/text.py:76-81`. No other code reads the patterns. (Verified via `grep -rn 'header_patterns' src/`.)

**Call site uses single-line strings.** `is_document_header(line, header_patterns)` is called at `extractors/glossary.py:482` and `:488` with `full_line_text = " ".join(s["text"] for s in line_spans)` — a single visual line, not a multi-page block.

**Regex flags.** `re.search(pattern, line, re.IGNORECASE)` — IGNORECASE only, NOT MULTILINE. So `^` anchors to start of the input string (the single line) and `$` to end. The patterns are effectively `fullmatch`-equivalent for single-line inputs. **MULTILINE blast-radius concerns (Codex #1) do not apply at the current call site.**

If a future change passes multi-line blocks to `is_document_header`, this contract breaks. Documented as a known coupling — separate maintenance burden if/when that happens.

## Hard 30%

### H1. False-positive risk on real glossary content (revised per Codex iter-1 #1)
The 2 new patterns are anchored `^\s*...\s*$` and applied via `re.search` with `re.IGNORECASE` only (no MULTILINE) against single-line strings — effectively `fullmatch` semantics. A real glossary entry's first line is the term, never `<TYPE> <pub#> Glossary-N`. Risk verified low.

### H2. Pattern variance across Army doctrine
Observed forms in the D-2 cohort:
- `"TC 3-22.240 Glossary-1"` ✓ caught by pattern 1
- `"13 January 2017 TC 3-22.6 Glossary-1"` ✓ caught by pattern 2
- `"ATP 3-21.10 Glossary-25"` ✓ caught by pattern 1

Possible un-covered variants:
- `"DA PAM 600-3 Glossary 5"` (space instead of dash) — covered by `[\s\-–—]+` separator.
- `"FM 3-34 GLOSSARY-1"` (all caps) — IGNORECASE flag is already applied at compile time, so handled.
- `"Glossary-1 ATP 3-21.10"` (order reversed) — NOT covered. Need to verify if this exists.

**Mitigation:** sample 5 D-2 cohort PDFs at execution time; verify all running-header variants are covered. If a new variant surfaces, add the pattern with a 1-line code comment.

### H3. Regression risk on 31-doc validation set
Some validation-set docs have running headers in the SAME format. v0.4 filtered them via `is_document_header` only when matching the existing patterns; the new patterns add coverage. Could a real validation-set doc have a line matching the new pattern that ISN'T a header? **Mitigation:** 31-doc golden-output regression. Any deviation gets `accepted-deviations.yaml` entry.

### H4. Header zone Y constraint (revised per Codex iter-1 #4)
`is_document_header` only fires for lines in `y_pos < HEADER_ZONE_Y` (150pt from top, `glossary.py:480-484`). Running-headers at >150pt slip through. **Mitigation:** verified empirically — the D-2 cohort's running-headers all appear at y ~50-100pt. **D-3-B does NOT change `HEADER_ZONE_Y`** — modifying that constant globally could suppress legitimate first-glossary-terms on pages where the first term starts near the top. If a future doc has running-headers below 150pt, that's a separate D-3-X unit (e.g., per-pattern-zone configuration), not a 1-line bump in D-3-B.

### H5. D-3-B-alone success semantics (per Codex iter-1 #5)
D-3-B's success criterion is **NOT entry recovery** — recovering acronym terms requires D-3-A too. D-3-B's success is:
- The running-header line `<TYPE> <pub#> Glossary-N` is correctly filtered before reaching the per-line term-classification gate.
- Verified by `is_document_header(line, header_patterns)` returning `True` on the running-header strings.
- Verified by absence of the running-header text in `validate_term` trace calls (via a small instrumented trace, not relying on D-2's full trace mechanism).

Entry recovery is tested SEPARATELY for combined D-3-A + D-3-B.

### H6. Regression gate (revised per Codex iter-1 #6)
`accepted-deviations.yaml` is NOT a free pass. For D-3-B specifically:
- Zero entry-count drops on the 31-doc validation set allowed.
- Zero valid-term removals.
- Any deviation must have documented root cause (likely: a previously-bogus running-header entry now correctly filtered).
- `definition_degradation` or `unexpected_removal` deviation categories (per D-3-A's harness extension) block merge.

## Pre-implementation: Step 1 — full running-header inventory (extended per Codex iter-2 #1+#3)

`scripts/v0.5-unit-d3b/inventory-headers.py` extracts every unique top-150pt-y-zone line containing "Glossary" or "GLOSSARY" from:
- All 7 P-2/hybrid docs in the D-2 cohort (target group).
- **All 31 docs in `validation_set/pdfs/`** (regression group — Codex iter-2 #1+#3).

Outputs:
- `validation_set/v0.5-unit-d3b-running-header-inventory.csv` with shape `{document_id, page_idx, y_pos, line_text, is_running_header, is_real_glossary_content, source: 'target_cohort' | 'validation_set'}`.

Each line is classified as `is_running_header=true` (matches one of the 2 proposed patterns) or `is_real_glossary_content=true` (genuine glossary line that happens to contain "Glossary"). Reviewer-confirmed pre-implementation.

**Step 1 acceptance:** zero lines classified as both `is_running_header=true` AND `is_real_glossary_content=true`. If any line trips both, the pattern set is wrong and the implementation does not proceed.

## Accepted open limitations (operator-judgment)

Per the iter-N convergence pattern observed across D-1, D-2, D-3-A: trajectory 8→12 on a 2-regex change indicates over-rigor for the work class. Lower-priority Codex iter-2 findings accepted as limitations:

- **#7 D-3-A harness dependency:** D-3-A's `categorized deviations harness extension` is a prerequisite. D-3-A lands first; D-3-B's regression gate inherits the categorized-deviations enforcement.
- **#8 Allowed positive/neutral deltas:** for D-3-B specifically, removing a previously-bogus running-header entry from the v0.4 golden output is `category=intended_removal_of_bogus_running_header` (a NEW deviation category added to the harness alongside D-3-A's `intended_acronym_addition`).
- **#9 Publication-number grammar precision:** `[\d\.\-–]+` is the empirically-validated grammar from Step 1's actual observed forms. If Step 1 surfaces an alphanumeric publication-id, broaden then; not preemptively.
- **#10 Army-only profile assumption:** only `ArmyProfile` exists today. Future profiles must re-audit. Documented in inline comment.
- **#11 Trace acceptance format:** not adding a separate trace; the integration test (#2 above) is the deterministic check.
- **#12 D-3-A + D-3-B combined validation:** combined-test happens at release-gate (separate plan when v0.5.0 ships).
