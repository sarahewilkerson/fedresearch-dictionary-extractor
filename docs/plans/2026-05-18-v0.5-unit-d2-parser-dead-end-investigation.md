# v0.5 Unit D-2: Parser Dead-End Investigation

**Date:** 2026-05-17
**Branch:** `feat/2026-05-18-v0.5-d2` (off post-merge main, post-D-1)
**Roadmap parent:** `docs/plans/2026-05-18-v0.5-roadmap-definition-extraction.md`
**Predecessor:** D-1 merged as commit `ed8ede5` (forward-scan-largest-block range detection)

## Context

After D-1 merged, the v0.5 residual zero-entry cohort consists of 16 docs:
- **11 Class-1 docs** — v0.4 already found the correct range; the parser then emitted 0 entries.
- **5 D-1-deferred docs** — under v0.5's forward-scan-largest-block algorithm these still produce the same range as v0.4 (their matching pages are all isolated single-page; no contiguous block exists; later-wins picks the same page v0.4's backward sweep picked). The range is "correct" in the same sense as Class-1: the parser then fails.

For all 16, **the range detector is working correctly. The parser is what's failing.** This unit is an **investigation**, not an implementation: characterize the parser failure modes, then propose follow-up units (one per distinct failure mode).

## Scope

D-2 ships **no production code changes** — `src/` is untouched. The unit DOES produce tooling (capture scripts, trace script) + investigation artifacts (cohort CSV, baseline JSON, per-doc traces, per-class fixtures) + a memo. Reframe from prior "read-only" wording per Codex iter-1 #1: tooling/fixture creation IS in-scope; production parser code is NOT.

In scope:

### Cardinality convention (Codex iter-2 #1)

This plan uses two named cohort sizes throughout:
- **`16`** — the original D-2 candidate set (11 Class-1 + 5 D-1-deferred).
- **`N`** — the verified parser-only cohort after Step 0 range-validation. `N ≤ 16`; the `16 - N` rerouted docs go to `validation_set/v0.5-d1-followup-candidates.yaml`.

All downstream verifiers reference `N`, not 16. Step 0's output `range-validation.csv` enforces `N + rerouted == 16` as a sanity check.

## Approach

For each of the 16 docs, run a deep parser trace and classify the failure mode. The investigation answers three concrete questions:

1. **Per-doc:** at what step does the parser dead-end? (no bold spans / acronym-line gate fails / column-detect fails / Section II narrowing eats everything / continuation cascade flushes term with empty def / gibberish filter rejects / etc.)
2. **Across the 16:** what failure-mode classes exist? Each class → one follow-up unit.
3. **Sample-page-level evidence:** for each failure mode, extract a fixture from one representative PDF showing the exact dead-end (term/def patterns the parser sees on that page).

### Investigation steps

**Step 0 — Range-correctness validation** (NEW per Codex iter-1 #2):
For each of the 16 docs, hand-verify that v0.5's detected range + Section II narrowed range actually corresponds to the real glossary section in the PDF. Methodology:
- Open the PDF, locate the actual glossary section (TOC or visual scan).
- Compare to v0.5's detected range and (if narrowing fired) narrowed range.
- Record per-doc: `range_correct: true|false`, `actual_glossary_range: [s,e]`, `notes`.
- **If a doc's detected range is WRONG, it's a D-1-followup, not D-2.** Re-route it: remove from cohort, add to a separate `validation_set/v0.5-d1-followup-candidates.yaml`. The remaining cohort is the verified parser-only failure set.
- Output: `validation_set/v0.5-unit-d2-range-validation.csv` with per-doc verdict.

**Step 1 — Establish baselines.** For each of the remaining (post-Step-0) docs, run `analyze_pdf` once and capture the full metadata + entry list (empty). Commit at `validation_set/v0.5-unit-d2-baseline.json`.

**Step 2 — Per-doc parser instrumentation.** Write `scripts/v0.5-unit-d2/trace-parser.py`. **Mechanism (specified per Codex iter-1 #3):**
- Use Python's `unittest.mock.patch` to wrap the internal helpers (`_validate_term`, `_looks_like_acronym_term_line`, `_flush`, `_filter_spans_to_below_header`, `_is_term_style_span`) WITHOUT changing their behavior — patches record `(args, return_value)` for every call, then return the un-patched function's actual result.
- Wrap `parse_glossary_entries` with a context manager that also records the per-line decisions in the outer loop via a thread-local trace collector (the loop itself is not modified; we instrument the helpers it calls).
- For decisions that happen inline (e.g., `is_new_term_line = in_term_col and (is_bold or is_acronym_line)`), reconstruct from helper call traces.
- **Validation** (Codex iter-1 #3): run instrumented + uninstrumented `parse_glossary_entries` on ALL 16 docs (not just one); assert identical entry count, identical entry contents (full term/def comparison), identical metadata. If divergence on any doc, the trace mechanism is faulty — fix before proceeding.

**Step 3 — Classification pass.** For each doc, apply the trace data through a classification rubric:
1. Did `find_glossary_page_range` return a result? (Should be yes for all 16 post-Step-0.)
2. Was Section II narrowing attempted? Did it fire?
3. How many lines passed `_validate_term`? How many got flushed?
4. Of flushed terms: how many had non-empty def_lines at flush? How many were dropped by `_flush`'s post-validation (gibberish, length, empty def)?
5. How many lines did the per-line gate accept as new-term-line?
6. Was the X-only fallback triggered?

**Open-ended classification (per Codex iter-1 #5):** assign each doc to an INDUCTIVELY-DERIVED class based on the trace pattern, NOT to the 5 hypothesized buckets. Hypotheses A-E are listed as priors only; if a doc doesn't fit any of A-E, it gets its own new class label. **Every doc that produces a trace gets a class assignment** — no anonymous failures.

**Step 4 — Representative fixture per class.** For each named class (post-Step-3), extract a fixture preserving the **full parser input shape** (Codex iter-1 #4):
- The PyMuPDF `page.get_text("dict")` output for the relevant glossary range page(s).
- Spans, bbox, font, flags (bold), text per span.
- Section narrowing context (which header pattern fired on which page).
- Serialized as JSON at `tests/fixtures/glossary_parser_v0.5/<class>-<doc-id>.json`.
- For classes with ≥3 docs, capture 2 fixtures from different docs (variance coverage). For classes with 1-2 docs, capture 1 fixture and flag the size limitation.

**Step 5 — Trace artifact preservation** (NEW per Codex iter-1 #7):
Commit a condensed per-doc trace summary at `validation_set/v0.5-unit-d2-trace-summaries/<doc-id>.txt`. Summary format:
- v0.5 metadata (range, section structure, narrowed range, fallback)
- Per-line trace counts (lines seen, term-candidates accepted, terms flushed, defs dropped, why)
- The first 20 lines of the actual trace (sample evidence)
- The classification verdict + rationale

This makes each classification auditable. Raw traces remain in `/tmp/d2-traces/` (too large to commit) but the summaries are the persistent evidence.

**Step 6 — Memo.** Write `docs/plans/2026-05-18-v0.5-unit-d2-findings.md` with the classification matrix + proposed follow-up units.

### Expected output classes (hypotheses — to verify, not assume)

Based on the v0.4 analysis from earlier in this session:

- **Class A (most likely):** OCR'd PDFs with no bold flags preserved AND title-case glossary terms (not all-caps). Bold gate fails; acronym gate fails; X-only fallback's all-lines-are-new-term cascade flushes everything with empty def lines.
- **Class B:** Section II narrowing produces an empty or single-page range (Class-1 had section_ii_only docs that may have hit this).
- **Class C:** Two-column glossary layouts that pymupdf's reading order interleaves wrong.
- **Class D:** Glossaries laid out as `term: definition` on a single line where the parser's term-column heuristic doesn't separate term from def.
- **Other:** unknown patterns surfacing in the trace.

Hypotheses are tentative. The investigation's job is to verify each empirically.

## Hard 30%

### H1. Trace instrumentation correctness (revised per Codex iter-1 #3 + iter-2 #3-#4-#8-#9)
**Trace contract:** the trace records per-line decisions including page, line-idx-in-page, bbox, first-span text, first-span bold flag, `in_term_col` decision, `is_bold` decision, `is_acronym_line` decision, `is_new_term_line` decision, term-text after multi-span walk, validation result + reason, flush trigger + def-state-at-flush, drop reason. This is the **trace contract** committed at `scripts/v0.5-unit-d2/trace-parser.py` docstring.

**Mechanism options (decided at execution time):**

Option A — **Helper wrapping** (preferred if sufficient): `unittest.mock.patch` wraps the named helpers (`_validate_term`, `_looks_like_acronym_term_line`, `_is_term_style_span`, `_flush`, `_filter_spans_to_below_header`); patches call through and record. Inline decisions (`in_term_col`, `is_new_term_line`) reconstructed from helper-call ordering.

Option B — **Non-production parser wrapper** (fallback if Option A insufficient): a `traced_parse_glossary_entries` function in `scripts/v0.5-unit-d2/trace-parser.py` that IS a near-copy of `parse_glossary_entries` with print/append statements added. **Required validation:** a `verify_trace_correctness.py` script (committed permanently per Codex iter-2 #8) that runs Option B's wrapper alongside the production parser on the entire N-doc cohort, asserts byte-identical entry list + metadata AND assert that the trace records ≥1 entry per non-empty input line. Both invariants must hold.

The mechanism choice is decided at investigation time and recorded in the memo with rationale.

**Validation beyond zero-entry equality (Codex iter-2 #4):** in addition to entry-count match, the validation script asserts:
- Number of lines fed to the parser equals the trace's line count.
- Every observed term-candidate has a recorded validation verdict.
- Every flushed term has a recorded def-state-at-flush.
- Every dropped def has a recorded drop reason.

## Accepted open limitations (operator-judgment)

The following Codex iter-2 findings are acknowledged but NOT fully resolved in the plan. They reflect a precision-vs-rigor tradeoff: this is an investigation unit producing a memo, not a production-traffic deliverable, and these items don't change the memo's substantive output:

- **Codex iter-2 #2 (range-correctness rubric).** Step 0 is hand-inspection with `notes` field; concrete page-index convention and "exact vs ±2" tolerance is documented in the memo when ambiguous cases surface, not pre-specified.
- **Codex iter-2 #6 (raw-trace preservation).** Raw traces are too large to commit; condensed summaries (committed) plus the documented capture command are the auditable artifact. Reviewers needing to re-run get the command, not the bytes.
- **Codex iter-2 #10 (range-validation evidence fields).** Step 0's CSV has `notes`; structured `pdf_page_label_start/anchor_text_start` etc. are not required pre-implementation. If a reviewer disputes a verdict, the operator opens the PDF.
- **Codex iter-2 #11 (cohort blob-hash assertion).** `source_artifact_commit: ed8ede5` in the CSV header is the provenance record. Adding a blob-hash assert against `git ls-tree` is over-engineering for a one-time investigation.
- **Codex iter-2 #12 (D-3-X regression controls).** The follow-up unit stubs include `blast_radius` but not pre-specified `negative_regression_fixtures` — those land at D-3-X plan time, not in this memo.

These limitations are documented here rather than worked around because: (a) D-2 ships no production code; (b) iter-1 → iter-2 trajectory was 8 → 12 findings (increasing), indicating full-gate criteria are over-rigorous for this scope; (c) the lighter-path heuristic at iter-1 flagged this correctly. The plan's value comes from the memo's classification, not from methodological perfection.
