## Class P-1: Section I acronym list filtered out by column-gate

**Pattern.** Docs with a Section I (Acronyms and Abbreviations) section where each entry is laid out as:

```
ADP   Army doctrine publication
ADRP  Army doctrine reference publication
AO    area of operations
```

The acronym (`ADP`, `ADRP`, `AO`) is indented to a column past `min_x + 30` because the OCR sees the section header `SECTION I — ACRONYMS AND ABBREVIATIONS` at an even-further-left position, becoming the page's `min_x`. So the term column is `[min_x, min_x + 30]`, the acronym lines start at `min_x + ~40` (just outside threshold) → `in_term_col = False` → not classified as new-term-line → continuation candidate — but `current_term` is None → silently dropped.

The `_looks_like_acronym_term_line` helper correctly identifies these as acronym-shaped (trace shows ~30 successful acronym detections per doc), but the X-column gate vetoes them BEFORE acronym-shape is consulted (the `is_new_term_line = in_term_col AND (is_bold OR is_acronym_line)` short-circuit).

**Evidence (fixture `p1-acronym-filtered.json`, ATP 1-05.01):** 30 `looks_like_acronym=True` events in the trace. Only 3 reached `validate_term`, all FAIL (on `"SECTION"` header text — invalid pattern). 0 actual acronym lines validated. 0 terms emitted.

**Affected docs (all 11; primary failure mode for 4).** Wherever Section I acronyms appear in this layout shape, the column-gate drops them.

## Class P-2: Page footer/header accepted as sole term

**Pattern.** Docs where the only line passing the term-column + bold/acronym gate is a page footer like:
- `"TC 3-22.240 Glossary-1"`
- `"13 January 2017 TC 3-22.6 Glossary-1"`

These appear at the bottom or top of each glossary page. Format: `"<doc-id> Glossary-<page-num>"`. The footer line passes `_looks_like_acronym_term_line` because `TC` is acronym-shaped (≤14 chars, all caps, valid acronym prefix). It then passes `_validate_term` because length is OK, not pure number, doesn't match an invalid pattern, no internal punctuation tripping the rules.

Once accepted as term, the parser awaits continuation lines. But the footer is the LAST line on its page, so `current_def_lines=[]` at end-of-page-flush → drop.

**Evidence (fixture `p2-footer-as-term.json`, TC 3-22.6):** 1 validate-pass on `"13 January 2017 TC 3-22.6 Glossary-1"`. 2 flush events, both dropped (empty def_lines).

**Root cause.** The existing footer-filter in `parse_glossary_entries` (PR1.2 Fix B at lines 487-505) catches footer matches inside `footer_y_threshold = page.rect.height * 0.88`. But these footers appear at the TOP of pages (running headers above the glossary content), outside the bottom-12% zone.

**Affected docs (primary failure mode for 1; contributor in 6 hybrids).**

## Proposed follow-up units (D-3-X)

### D-3-A: Column-gate fallback for Section-I-style acronym lists

**Root cause:** Class P-1. The `is_new_term_line = in_term_col AND (is_bold OR is_acronym_line)` gate's column-check is too strict when the page has a section header (Section I title) much further left than the actual acronym entries. `min_x` is dominated by the section header.

**Affected doc IDs:** all 11 verified parser-dead-end docs (primary or contributing).

**Fixture coverage:**
- Positive (must recover): `tests/fixtures/glossary_parser_v0.5/p1-acronym-filtered.json`
- Negative (must not break): D-1's `validation_set/v0.5-unit-d1-v04-golden-output.json` (31-doc golden)

**Intended parser surface:** `parse_glossary_entries` in `src/fedresearch_dictionary_extractor/extractors/glossary.py` — specifically the `term_col_threshold` computation or the `is_new_term_line` gate.

**Approach (sketch):** when `min_x` would come from a section-header line (heuristic: line text matches `SECTION` pattern), exclude it from `min_x` computation. Alternatively: relax the gate so `is_acronym_line=True` overrides `in_term_col=False`, with safety check that the acronym line's first_x is within a "reasonable" range (e.g., `min_x + 60` instead of `min_x + 30`).

**Non-goals:** changing the bold-gate. Changing `_validate_term`. Changing the X-only fallback trigger (already covered by `_bold_preservation_rate < 0.10` check).

**Blast radius:** the 6 hybrid docs (P-1+P-2) may emit acronym entries — that's the goal. The 4 P-1-only docs may emit. The 31-doc golden set could be affected if any of those docs have acronyms previously filtered (NEGATIVE REGRESSION RISK). Mitigation: full golden-output run before merge.

**Verification criteria:**
- `pytest tests/test_glossary_parser_fixtures.py::test_fixture_reproduces_zero_entries[p1-acronym-filtered]` — must now FAIL (entries emitted = good).
- New test: `test_p1_fixture_recovers_at_least_N_acronyms` (assert ≥10 emitted entries from p1 fixture).
- `test_validation_set_golden_output_stable` from D-1 — must still pass for all 31 (no new regressions OR each new deviation in accepted-deviations.yaml).

### D-3-B: Top-of-page running-header filter

**Root cause:** Class P-2. Page footers/running-headers like `"TC 3-22.240 Glossary-1"` slip through the existing footer-zone filter because they appear at the TOP of the page (not in the bottom-12% zone).

**Affected doc IDs:** the 7 docs in P-2 or P-1+P-2 hybrid.

**Fixture coverage:**
- Positive (must recover): `p2-footer-as-term.json` AND `p1-p2-hybrid.json`
- Negative (must not break): 31-doc golden

**Intended parser surface:** `parse_glossary_entries` — extend the footer-zone filter (lines 487-505) to also check a TOP zone (e.g., `y_pos < page.rect.height * 0.05`), OR add a regex to `profile.header_patterns` matching the `<TYPE> <pub#> Glossary-<N>` shape.

**Approach (sketch):** add a new regex to `ArmyProfile.header_patterns` (already used by `is_document_header` filter):
```python
r"^(AR|FM|ADP|ATP|TC|PAM|TM|DA\s*PAM)\s+[\d\.\-]+\s+Glossary[\s\-–—]+\d+",
r"^\d{1,2}\s+\w+\s+\d{4}\s+(AR|FM|ADP|ATP|TC|PAM|TM)\s+[\d\.\-]+\s+Glossary[\s\-–—]+\d+",
```

**Non-goals:** changing the bottom-footer filter. Changing term-column logic.

**Blast radius:** any doc with a running-header matching the new regex will have those lines filtered. Real glossary content shouldn't match (the running-header regex is whole-line anchored). Mitigation: full golden-output run.

**Verification criteria:**
- `pytest tests/test_glossary_parser_fixtures.py` — p2 + hybrid fixtures must now emit ≥1 entry.
- `test_validation_set_golden_output_stable` — no new regressions.
- New test: `test_running_header_pattern_rejects_actual_glossary_terms` (verify pattern doesn't false-positive on real glossary terms by feeding ~50 known-good terms through the regex).
